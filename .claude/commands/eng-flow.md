# Engineering Delivery Flow

You are the orchestrator for the closed-loop engineering delivery flow. The flow
covers requirement readiness, solution design, BDD/test design, implementation,
CI/CD gates, scripts/automation, release readiness, production launch, and post-launch
closure.

**Input:** `$ARGUMENTS`

---

## Step 1: Parse Arguments

Determine the source and requested stage from `$ARGUMENTS`.

Supported source forms:

- **JSON file mode:** argument starts with `/`, `./`, `~`, or ends with `.json`
- **ADO mode:** argument is a pure number, starts with `ado:`, or contains
  `dev.azure.com`
- **Resume mode:** argument is empty or contains `resume`

Supported stage flags:

- `--stage readiness`
- `--stage solution-design`
- `--stage bdd`
- `--stage delivery-design`
- `--stage design` (alias for `delivery-design`)
- `--stage scripts`
- `--stage implement`
- `--stage ci`
- `--stage acceptance`
- `--stage release`
- `--stage launch`
- `--stage close`
- `--from {stage}` and `--to {stage}` for bounded execution

If input is empty and no resumable state exists, ask:

> "Please provide a source: a JSON story file, an ADO Work Item ID/URL, or an existing `.claude/engineering-flow.local.md` state to resume."

---

## Step 2: Load Source And State

Read `.claude/engineering-flow.local.md` if it exists. Preserve existing entries.

### If JSON file mode

Use the Read tool to read the JSON. Prefer the confirmed Story Contract JSON
generated by `/writeuserstories` and defined in `~/.claude/docs/story-contract.md`,
or a generic story schema with these fields:

```json
{
  "storyId": "12345",
  "title": "...",
  "description": "...",
  "acceptanceCriteria": [
    {
      "id": "AC-001",
      "given": "...",
      "when": "...",
      "then": "..."
    }
  ],
  "persona": "...",
  "module": "...",
  "nonFunctionalRequirements": [],
  "dependencies": [],
  "releaseTarget": "...",
  "riskLevel": "medium"
}
```

For legacy OREO schema or raw story payloads, run `/writeuserstories` first. `/eng-flow` and `/bdd-gen` do not own story normalization.

### If ADO mode

Extract the Work Item ID:

- Pure number -> use directly
- `ado:{id}` -> extract number after `ado:`
- Full URL -> extract the ID from the URL path

Use the `ado-agent` MCP tool when available to fetch the Work Item details. Do
not use curl/PAT unless the user explicitly asks for that fallback.

### If Resume mode

Use the latest incomplete story section in `.claude/engineering-flow.local.md`.
If multiple stories are incomplete, ask the user which story to resume.

---

## Step 3: Normalize Requirement Context

Extract and normalize:

| Field | Source |
|-------|--------|
| storyId | JSON `storyId` or ADO Work Item ID |
| title | JSON title or ADO title |
| description | JSON/ADO description |
| acceptanceCriteria | Given/When/Then objects from the Story Contract |
| persona | explicit field or infer from description |
| storyModule | JSON module or ADO scope/module |
| nonFunctionalRequirements | explicit field or parsed notes |
| dependencies | explicit field or parsed links/notes |
| releaseTarget | explicit field or unknown |
| riskLevel | explicit field or infer from scope/dependencies |

Show the normalized story summary to the user and request one of:

- **approve** - proceed to requirement readiness
- **revise** - modify normalized fields
- **stop** - terminate flow

## Step 5: Solution Design Enrichment Gate

If the Story Contract already contains `solutionDesign.designStatus: approved`,
summarize the design context and ask whether to reuse or revise it.

If not present or not approved, run `/enrichstorydesign` with the confirmed
Story Contract JSON. The design enrichment must clarify relevant API contracts,
UI states, data rules, permissions, integrations, non-functional constraints,
observability, rollout, and test design implications.

When design enrichment is complete, capture:

- API operations and response contract references
- UI entry points, visible states, validation messages, dialogs, and accessibility notes
- Data, integration, security, NFR, observability, and rollout constraints
- Test design implications, including API contract files, UI observable evidence,
  extra API assertions, and test data needs
- Open design questions

Ask the user:

- **approve** - proceed to BDD/test design
- **revise** - adjust solution design or rerun enrichment
- **stop** - terminate flow

Update `.claude/engineering-flow.local.md` with current stage
`solution-design`.

---

## Step 6: BDD And Test Design Gate

If BDD artifacts are already present and current, summarize them and ask whether
to reuse or regenerate.

If not present, run the existing BDD generation pipeline:

1. Follow `.claude/commands/bdd-gen.md` using the design-ready Story Contract JSON when available.
2. Ensure Phase 1 layering analysis is approved.
3. Ensure Phase 2 feature generation is approved.
4. Write approved feature files.
5. Update story state with generated feature paths and TC tags.

When BDD generation is complete, capture:

- API feature file paths
- UI feature file paths
- Scenario count and TC tags
- AC coverage matrix
- New snippet or Java step needs

Ask the user:

- **approve** - proceed to delivery design
- **revise** - adjust test design or feature files
- **stop** - terminate flow

Update `.claude/engineering-flow.local.md` with current stage `bdd-design`.

## Step 8: Scripts And Automation Gate

Scripts are a first-class delivery artifact. Before implementation or CI, build
a script inventory and decide whether any missing automation must be created.

Read `.claude/docs/engineering-scripts.md`, then discover existing automation:

- Repository guidance: `README*`, `AGENTS.md`, `CLAUDE.md`, `CONTRIBUTING*`
- Build/test entry points: `package.json`, `Makefile`, `justfile`,
  `Taskfile*`, `pom.xml`, `build.gradle`, `gradlew`, `pyproject.toml`,
  `tox.ini`, `noxfile.py`
- Local scripts: `scripts/`, `bin/`, `tools/`, `dev/`
- CI/CD config: `.github/workflows/`, `.gitlab-ci.yml`, `.harness/`,
  `harness/`, `.azure-pipelines/`, `azure-pipelines.yml`, `Jenkinsfile`
- Deploy/runtime config: Helm, Kustomize, Terraform, Docker Compose,
  Kubernetes manifests, environment overlays

If available, run:

```bash
scripts/engineering-flow/discover-automation.sh .
```

Use its markdown output as the initial Script Inventory, then adjust based on
the Delivery Design Report and risk level.

Produce a Script Inventory:

| Gate | Existing Script/Command | Status | Gap | Action |
|------|-------------------------|--------|-----|--------|
| Local verification | | ready/missing/needs update | | |
| CI quality gate | | ready/missing/needs update | | |
| BDD/API/UI acceptance | | ready/missing/needs update | | |
| Non-prod deploy | | ready/missing/needs update | | |
| Smoke | | ready/missing/needs update | | |
| Release readiness | | ready/missing/needs update | | |
| Production launch | | ready/missing/needs update | | |
| Rollback | | ready/missing/needs update | | |
| Evidence capture | | ready/missing/needs update | | |

Script decisions:

- If required scripts already exist, reuse them and record the command.
- If a script is missing but not required for the current risk level, record it
  as a non-blocking gap.
- If a script is required, ask for approval before creating or changing it.
- Do not create production launch or rollback scripts without explicit user
  approval and a clear environment-safety contract.
- Any new script must follow `.claude/docs/engineering-scripts.md`.

Display the Script Inventory and ask:

- **approve** - proceed to implementation
- **revise** - adjust script inventory or create/update approved scripts
- **stop** - terminate flow

Update `.claude/engineering-flow.local.md` with current stage
`scripts-automation`, inventory, and any approved script work.

---

## Step 9: Implementation

Only proceed after delivery design and scripts/automation approval.

Implementation rules:

1. Inspect the repository before editing.
2. Keep changes scoped to approved implementation slices.
3. Preserve unrelated user changes.
4. Add or update tests according to the Delivery Design Report.
5. Update docs/config only when needed for the change.
6. Record changed files in state.

Before editing, state what files/modules will be changed and why.

After implementation, run the local verification commands from the Delivery
Design Report. If commands are missing, infer the safest project-local commands
from the Script Inventory and repository conventions, then report the
assumption.

Update `.claude/engineering-flow.local.md` with current stage `implementation`,
changed files, and local verification results.

---

## Step 10: CI Quality Gate

Run or inspect required CI checks.

Expected evidence:

- Build result
- Unit/integration/API/UI test results as applicable
- Lint/type/static analysis results
- Security scan results when required
- Artifact version or commit SHA
- Script or pipeline step used for each gate

If Harness CI/CD is configured and the user asks to trigger/check it, use the
available integration or CLI. Do not trigger production-affecting pipelines
without explicit approval.

If CI cannot be run locally or no integration is available, produce a CI
Evidence Gap section with exact commands or pipeline links the user must run.

Update `.claude/engineering-flow.local.md` with current stage `ci-gate` and
evidence links/results.

---

## Step 11: Environment Acceptance

Deploy or verify deployment to non-production only when a safe command or
pipeline is available and approved.

Run acceptance checks:

- Generated BDD scenarios or equivalent API/UI checks
- Smoke tests for changed service/module
- Regression tests required by the Delivery Design Report
- Manual QA/PO checks if marked required

Record:

- Environment
- Version/commit
- Test commands and results
- Defects and severity
- Acceptance decision
- Smoke/deploy scripts or equivalent pipeline steps used

Update `.claude/engineering-flow.local.md` with current stage `env-acceptance`.

## Step 13: Production Launch

Proceed only with explicit user approval.

Launch rules:

1. Confirm target environment is production.
2. Confirm release strategy and rollback trigger.
3. Trigger only the approved deployment, script, pipeline, or feature flag
   action.
4. Capture deployment ID, version, timestamp, and owner.
5. Run production smoke checks using the approved smoke script, pipeline step,
   or runbook.
6. Watch required health signals for the agreed window.

If a rollback trigger fires, stop rollout, execute or recommend rollback based
on available approval, and move to post-launch closure with
`Rollback Follow-Up`.

Update `.claude/engineering-flow.local.md` with current stage
`production-launch`.
