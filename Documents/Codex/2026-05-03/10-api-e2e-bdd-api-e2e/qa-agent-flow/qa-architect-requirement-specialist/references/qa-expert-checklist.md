# 专家 Checklist 详解 (QA Expert Checklist)

分析需求时，**必须**针对以下四类「隐性需求」做强制检查，并在【风险建议/缺陷预警】中体现结论。

---

## 1. 幂等性检查 (Idempotency)

**关注点**：同一操作多次执行（连续点击、重试、重复提交）是否会导致数据重复或重复扣减？

**典型场景**：
- 提交订单、支付回调、库存扣减、发券、写入日志。
- 前端重复点击、网关/客户端重试、MQ 重复消费。

**检查方式**：
- 需求中是否明确「幂等键」或「唯一业务号」设计？
- 写操作是否有「先查再写」或「唯一约束 + 忽略重复」等防重逻辑？
- 扣减类操作是否采用「预占 + 确认」或版本号/乐观锁？

**输出建议**：在测试点中增加「重复提交/重试」用例；在风险建议中标注未体现幂等设计的接口与风险等级。

---

## 2. 一致性风险 (Consistency)

**关注点**：主从库同步延迟、缓存与 DB 不一致、缓存击穿/穿透对业务的影响。

**典型场景**：
- 读写分离下，写后立即读从库可能读不到最新数据。
- 缓存失效、缓存与 DB 双写顺序不当导致脏读。
- 高并发下缓存击穿导致 DB 压力激增。

**检查方式**：
- 需求是否区分「强一致读」与「最终一致读」？
- 是否有缓存更新策略（Cache-Aside、双写、过期时间）说明？
- 关键路径是否考虑主从延迟时间窗内的用户行为？

**输出建议**：在架构级测试点中列出主从延迟、缓存一致性场景；在风险建议中提示一致性边界与可接受延迟。

---

## 3. 兼容性深度 (Compatibility)

**关注点**：旧版数据如何适配新字段？存量业务是否能无损升级与回滚？

**典型场景**：
- 新增必填字段：老数据为 null 时展示与逻辑是否兼容？
- 枚举扩展：老数据旧枚举值在新逻辑下是否仍合法？
- 接口字段新增/删除：老版本客户端是否仍可工作？
- 配置/开关变更：灰度与回滚方案是否明确？

**检查方式**：
- 表结构/接口变更是否标注「兼容性说明」或「数据迁移/回填」方案？
- 是否有「默认值」「兼容枚举」「废弃但保留」的设计？
- 版本化与灰度发布是否在需求或迭代范围内？

**输出建议**：在测试点中增加「老数据 + 新逻辑」「老客户端 + 新接口」类用例；在风险建议中列出兼容性盲区与迁移建议。

---

## 4. 可观测性 (Observability)

**关注点**：业务埋点与日志是否完整？是否足以支撑线上快速排障与复盘？

**典型场景**：
- 关键业务节点（下单、支付、状态变更）是否有唯一 traceId 串联？
- 异常与错误是否有分级日志与告警？
- 核心指标（成功率、延迟、业务量）是否有监控与大盘？

**检查方式**：
- 需求或技术方案中是否提及日志规范、traceId、关键埋点？
- 是否明确哪些异常需要告警、哪些仅记录？
- 是否有「排障时需能查到的信息」的隐含要求？

**输出建议**：在风险建议中补充「可观测性」缺口（如缺少某环节埋点、日志字段不足）；必要时在测试点中增加「日志与 trace 校验」类用例。

---

## 使用方式

在完成「深度解析 + 技术风险预判 + 测试点设计」后，按上表逐项过一遍，将结论写入输出的**【风险建议/缺陷预警】**，并对未覆盖项给出优先级与改进建议。
