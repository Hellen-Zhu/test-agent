# `.bdd-step-index/` — Built-in step discovery index

This directory holds a pre-extracted, agent-readable index of every step (`@Given` / `@When` / `@Then`) defined in the team's internal BDD framework Maven dependencies (e.g. `com.hellen.framework.bdd:core-steps`, `:web-steps`, …).

It exists so that BDD-authoring agents (e.g. the `cucumber-e2e-automation` skill) can perform **Tier 1 step discovery** — see `bdd-cucumber-skills/.claude/skills/cucumber-e2e-automation/references/step-discovery.md` — using only Read / Glob / Grep tools, with no Bash and no Maven invocation at runtime.

## Why this exists (the enterprise rationale)

Agents that author Cucumber step definitions must verify that a built-in step doesn't already exist *before* they write a new one. The naive way to do that is to have the agent `unzip` the sources JAR from `~/.m2` and `grep` for annotations — but that requires shell access, which expands the agent's blast radius to "anything Bash can do".

Pre-extracting the index at build time keeps the agent's tool surface minimal (Read / Glob / Grep only) while preserving the reuse check. The index file is human-diffable, version-pinned to the JAR it was generated from, and reviewable in PRs.

| Property | Value |
|----------|-------|
| Generated by | `step-index-maven-plugin` (framework team's Maven plugin) |
| Generated when | `mvn install` (binds to `generate-test-resources` phase) |
| Consumed by | BDD-authoring agents at runtime, via Glob + Grep |
| Source of truth | The locked dependency JAR — `pom.xml` resolves the version |
| Committed to git | **No** — the README is committed; the `*.txt` index files are gitignored |

## File layout

```
.bdd-step-index/
├── README.md              ← committed; explains the convention
├── builtin-steps.txt      ← all built-ins, pipe-delimited (generated, gitignored)
├── auth-steps.txt         ← optional shards by area (generated, gitignored)
├── form-steps.txt
└── ...
```

## Line format

Each line in a `*.txt` index file is pipe-delimited with three fields:

```
<annotation>|<step regex>|<fully-qualified method reference>
```

Example:

```
@When|I click {string}|com.hellen.framework.bdd.steps.FormSteps#click
@When|I fill {string} with {string}|com.hellen.framework.bdd.steps.FormSteps#fillField
@Then|I should see {string}|com.hellen.framework.bdd.steps.AssertionSteps#shouldSeeText
```

Why this shape:

- **Greppable** — `grep '^@When|I click '` returns one line per match.
- **Field 3 is the audit anchor** — reviewers can click through to the FQN in the sources JAR if they want to verify.
- **No JSON / no YAML** — the index is a data feed, not a config. Keep it boring.

## Regeneration cadence

The index regenerates automatically whenever you run any Maven phase that includes `generate-test-resources` (which is all the usual ones: `compile`, `test`, `install`, `verify`, …). You should never need to edit these files by hand.

If the index goes stale (you bumped the framework dep but didn't run a build), agents will read the old version and may miss newly-added steps. Failure mode: a `NEW_STEP` slips through review for a step that actually exists in the new dep version. Mitigation: CI runs `mvn install` (which regenerates the index) before any agent-authored PR is opened for review.

## When this directory is empty / missing

A fresh checkout will not have any `*.txt` files until `mvn install` runs at least once. Agents are taught to handle this: if Tier 1 (index) is unavailable, they walk Tiers 2–5 (catalog doc → cached Javadoc → AskUserQuestion → mark unavailable) — see `step-discovery.md` for the full hierarchy.

If the Maven plugin itself isn't wired into the project's `pom.xml` yet, that's a build-engineer follow-up — see `step-discovery.md § Bootstrapping .bdd-step-index/` for the plugin configuration.

## Governance notes

- The README in this directory **is committed** — it documents the convention for humans.
- The `*.txt` index files **are gitignored** — they're build outputs, like `target/`.
- The Maven plugin that generates them is owned by the **framework team**, not the test-authoring team. If the plugin is broken or the JAR layout changes, raise upstream with that team rather than working around it in the agent.
- Do not add agent-authored content to this directory. Agents read it; only the build writes to it.
