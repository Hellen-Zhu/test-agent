# 需求分析阶段核心能力矩阵 (Expertise Matrix)

根据资深 QA 与测试开发能力标准，需求分析阶段应具备以下 5 大核心能力，并在「深度解析 → 技术风险预判 → 测试点设计」工作流中综合运用。

---

## 1. 业务与上下文解构 (Business & Context Decomposition)

- **目标**：理解功能在整体业务中的位置，以及对存量系统的影响。
- **关键动作**：
  - 识别该需求影响的模块、表、接口、配置范围。
  - 判断对存量业务的侵入性：是否改表结构、枚举、兼容老数据。
  - 明确上下游依赖与调用方，避免「单点需求」导致链路盲区。
- **输出**：业务全景图、影响范围说明、存量兼容策略。

---

## 2. 角色与权限分析 (Role & Permission Analysis)

- **目标**：厘清多角色场景下的操作边界与数据可见性，避免越权与逻辑漏洞。
- **关键动作**：
  - 列出所有参与角色（如医生、分诊员、管理员、患者）。
  - 定义每个角色可访问的数据范围、可执行的操作、不可见/不可操作项。
  - 区分垂直越权（跨角色）与水平越权（同角色跨数据）。
- **输出**：角色-权限矩阵、越权测试关注点。

---

## 3. 状态与生命周期 (State & Lifecycle)

- **目标**：保证业务对象状态机闭环，无孤儿状态与非法迁移。
- **关键动作**：
  - 梳理核心实体的状态集合与初始/终态。
  - 明确每个迁移的触发条件、前置状态、后置状态。
  - 检查是否存在不可达状态、无法回滚或无法补偿的路径。
- **输出**：状态机图/表、合法与非法迁移清单。

---

## 4. 技术风险与数据流 (Technical Risk & Data Flow)

- **目标**：从架构与数据链路层面识别一致性、性能、可用性风险。
- **关键动作**：
  - 绘制/描述数据流：前端 → 网关 → 服务 → MQ → 消费端 → 缓存 → DB。
  - 标出读写分离、主从延迟、缓存失效对读写的影响。
  - 识别高频接口与热点数据，预判并发与容量边界。
  - 梳理回滚、熔断、降级与补偿逻辑是否在需求中体现。
- **输出**：数据流说明、性能/一致性风险点、异常流设计建议。

---

## 5. 测试点与质量保障设计 (Test Point & Quality Assurance Design)

- **目标**：将上述分析转化为可执行的测试点，覆盖功能、架构、性能、安全。
- **关键动作**：
  - 功能：颗粒度到字段校验、关联约束、状态迁移，含 Happy Path、边界、逆向。
  - 架构：接口契约、数据一致性、索引与慢查询、缓存与 DB 一致性。
  - 性能：TPS/RT 目标、压测场景、资源水位。
  - 安全：脱敏、垂直/水平越权、敏感操作审计与合规。
- **输出**：深度测试点清单（按架构级/业务级/性能安全分类）、优先级建议。

---

## 运用顺序

在单次需求分析中，建议顺序为：**1 → 2 → 3 → 4 → 5**，并在步骤 4、5 中结合专家 Checklist（幂等性、一致性、兼容性、可观测性）做隐性需求检查。
