# 标准输出模板 (Standard Output Template)

需求分析完成后，**必须**按以下结构组织输出，四部分缺一不可。

**双输出约定**：完成需求分析后，需同时产出 **(1) 需求分析完整文档** 与 **(2) 开发用需求测试点 Checklist**。开发用 Checklist 模板见 [dev-checklist-template.md](dev-checklist-template.md)。

**图片输入时的整理方式**：若输入为多张需求相关图片，文档内容应**按图片序号（图1、图2、…）组织**：在【逻辑与状态机】【深度测试点】等章节中，按图号分小节或分条标注来源，使每条分析可对应到具体图片，便于评审与追溯。

---

## 1. 【业务全景图】

- **定义**：用一句话说清该需求对应的核心商业逻辑与业务价值。
- **要求**：面向产品/研发/测试都能快速理解「在做什么、为什么做」。
- **示例**：  
  「本需求实现「预约挂号锁号-支付-取号」闭环，用户在选定时段完成支付后，在取号窗口凭码取号，核心价值是减少现场排队并提高号源利用率。」

---

## 2. 【逻辑与状态机】

- **定义**：梳理核心业务对象的状态集合、迁移条件及合法/非法路径。
- **要求**：
  - 列出主要状态节点（如：待支付、已支付、已取号、已取消、已过期）。
  - 标明触发条件（用户操作、定时任务、外部回调等）。
  - 标出非法迁移（如：已取消不可再支付）及边界（如：超时自动取消）。
- **形式**：表格或简要状态图描述均可，确保无「孤儿状态」与无法迁移的中间态。

**示例表格**：

| 当前状态 | 可迁移至     | 触发条件           |
|----------|--------------|--------------------|
| 待支付   | 已支付/已取消/已过期 | 支付成功/用户取消/超时 |
| 已支付   | 已取号       | 用户取号           |

---

## 3. 【深度测试点】

按以下三个子维度组织，每条测试点需可执行、可判定。

### 3.1 架构级关注

- 接口契约：入参/出参、错误码、超时与重试约定。
- 数据一致性：主从延迟、缓存与 DB 一致性、双写顺序。
- 数据库与存储：索引是否支持查询条件、是否存在慢查询与锁表风险。

### 3.2 业务级关注

- Happy Path：从入口到终态的主流程，按角色与状态分条。
- 边界：字段长度、取值范围、数量上限、时间窗口边界。
- 逆向与异常：取消、超时、失败回滚、部分成功与补偿。

### 3.3 性能/安全关注

- 并发与压测：高频接口、预期 TPS/RT、大促或高峰场景。
- 权限：垂直越权（跨角色）、水平越权（同角色跨数据）。
- 脱敏与审计：身份证/手机号等脱敏策略、敏感操作是否留痕与可审计。

---

## 4. 【风险建议/缺陷预警】

- **定义**：针对需求文档中的**模糊点、矛盾点、遗漏点**提出挑战与改进建议。
- **要求**：
  - 每条建议对应具体需求描述或模块，并说明「风险」或「可能缺陷」。
  - 可结合专家 Checklist（幂等性、一致性、兼容性、可观测性）输出结论。
  - 可标注优先级（高/中/低）与建议归属（需求澄清/设计补充/测试重点关注）。
- **示例**：
  - 「支付回调未明确幂等键，重复回调可能导致重复核销，建议需求中补充幂等设计与测试用例。」
  - 「读写分离下「支付成功页」立即查订单可能读从库未同步，建议标注强一致读或可接受延迟时间窗。」

---

## 使用说明

- 若需求信息不足，在对应章节注明「待产品/研发确认」，并在【风险建议/缺陷预警】中列出待澄清项。
- 输出中应能体现「深度解析 → 技术风险预判 → 测试点设计」的推导过程，以及专家 Checklist 的检查结论。
- **多图输入**：在【逻辑与状态机】【深度测试点】中按图号分节（如「图1-登录页」「图2-列表页」），保证完整文档可与图片一一对应。
- **同步输出**：在交付本需求分析文档的同时，按 [dev-checklist-template.md](dev-checklist-template.md) 输出「开发用需求测试点 Checklist」，便于直接给开发使用。
